01 · 에이전트 루프
단순한 핵심,
복잡한 주변

루프 자체는 작습니다. 그 주변의 인프라가 기능이 늘어도 안정성을 유지하게 해줍니다.

  • 동작하는 에이전트 루프는 약 20줄의 코드로 충분합니다.
  • 제어 흐름은 분기하는 내부 상태가 아닌 도구에 있습니다.
  • 워크플로 vs 에이전트: 코드에서 경로를 미리 정하느냐, 실행 시에 모델이 경로를 선택하느냐의 차이.
루프가 스프린트마다 커지고 있다면 잘못된 레이어를 고치고 있는 것입니다. 비용은 루프 바깥에서 발생합니다.

02 · 하네스
하드웨어보다
하네스가 이긴다

더 비싼 모델로 얻는 이득은 예상보다 훨씬 작습니다. 검증, 경계, 피드백, 폴백이 모델 능력보다 더 중요합니다.

  • 하네스를 먼저 개선하세요. 정확도가 움직이지 않는다면, 그때 모델을 바꾸세요.
  • 평가는 유일하게 정직한 신호입니다. 모델 테스트 전에 하네스를 테스트하세요.
  • 작은 모델 + 강한 하네스가 큰 모델 + 약한 하네스를 프로덕션에서 반복적으로 이깁니다.

03 · 컨텍스트
길이보다 밀도

긴 컨텍스트 윈도우는 약한 컨텍스트 설계를 고쳐주지 않습니다. 컨텍스트 부패는 모델에 상관없이 30~40만 토큰쯤에서 시작됩니다.

  • 로드를 계층화하세요: 상수, 온디맨드, 런타임, 메모리, 시스템.
  • 먼저 인덱싱하고, 필요할 때 전체 내용을 가져오세요. 처음부터 전부 쏟아붓는 것보다 효과적입니다.
  • 안정적인 프롬프트 접두어는 캐싱 효과를 실질적으로 높입니다.
  • 로드를 지지하지 않는 모든 토큰은 신호를 희석합니다.

04 · 도구 & 메모리
상태를 컨텍스트
밖으로 꺼내라

도구는 기저 API 형태가 아닌 에이전트의 목표에 맞게 설계해야 합니다. 메모리는 윈도우가 아닌 디스크에 있어야 합니다.

  • ACI 원칙: 에이전트-컴퓨터 인터페이스 — HTTP 동사가 아닌 에이전트가 하고 싶은 것을 위해 설계하세요.
  • 나쁜 도구 설명은 도구 설명을 다시 읽기 전까지는 모델 실패처럼 보입니다.
  • 파일 기반 상태는 재시작 후에도 살아남습니다. 컨텍스트 내 상태는 그렇지 않습니다.

네 가지 메모리 유형

  • 작업 기억: 컨텍스트 윈도우 — 빠르고, 비싸고, 일시적입니다.
  • 절차적 기억: SKILL.md 파일 — 어떻게 행동할지, 지연 로드됩니다.
  • 에피소딕 기억: JSONL 로그 — 무슨 일이 있었는지, 추가 가능합니다.
  • 의미적 기억: MEMORY.md — 세션을 넘어 무엇을 기억할지, 임계점에서 통합됩니다.
세션 간 일관성은 의도적인 통합이 필요합니다. 기대에 맡겨서는 안 됩니다.

05 · 코드 스타일
문법보다 의사코드

주석이 코드 줄보다 많아야 합니다. 독자는 언어 튜토리얼이 아닌 논리를 봅니다.

  • 의도를 먼저 쓰고, 그 아래에 구현 세부 사항을 적으세요.
  • 변수명은 타입이 아닌 그것이 무엇인지를 설명합니다.
  • 블록당 하나의 개념. 과감하게 분리하세요.
# 도구 호출을 처리하거나 중단을 결정 function agent_step(context, tools): # 모델이 다음 행동을 선택 action = model.think(context) # 종료 조건: 모델이 완료라고 판단 if action.type == "stop": return action.result # 적절한 도구에 위임 result = tools[action.name](action.args) return agent_step(context + result, tools)
